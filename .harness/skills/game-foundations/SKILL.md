# 游戏开发元原则

## 1. 多分辨率适配（硬约束）

**所有核心UI元素必须在不滚动的窗口内可见。** 支持 2560x1440 作为基准，但必须向下兼容到 1920x1080。

### 布局策略

- 使用**比例定位**而非硬编码像素坐标。例如：`sidebar_x = screen_width * 0.85` 而非 `sidebar_x = 2200`
- 使用 `screen.get_width()` 和 `screen.get_height()` 动态获取实际窗口尺寸
- 所有坐标计算基于这些动态值，不写死 2560 或 1440
- 字体大小按比例缩放：`font_size = int(screen_height / 45)` 而非 `font_size = 32`

### 必须适配

- 窗口 resize 事件处理
- 全屏/窗口切换
- 最小支持分辨率：1920x1080
- 当分辨率不足时，优先保证核心信息可见，次要装饰可裁剪

### 固定UI vs 可滚动世界

游戏画面分为两层：

**固定UI层（HUD）** — 始终钉在屏幕上，不随地图移动：
- 阶段/回合指示器、倒计时
- 玩家列表、操作按钮、日志面板
- 使用屏幕坐标定位，不受地图滚动影响

**可滚动世界层（Camera）** — 玩家可拖动/滚轮查看：
- 村庄地图、房屋、装饰物、道路
- 世界坐标 > 视口时，通过 camera offset 平移
- 支持鼠标拖拽平移 + 滚轮缩放
- Camera 有边界限制，不能拖出世界范围

```python
# Camera 模式示例
camera_x = 0  # 世界偏移量
camera_y = 0
# 世界坐标 → 屏幕坐标：world_x - camera_x, world_y - camera_y
# 拖动时更新 camera_x/y，限制在世界范围内
```

这样即使地图比视口大，核心 UI 始终可见，玩家可以自由探索大地图

### 当前问题

目前代码中大量硬编码坐标（如 `SIDEBAR_X = 2260`, `offset_left = -300`），导致在非 2K 屏幕或窗口化时元素溢出。必须逐步替换为比例计算。

## 2. 完整游戏规则

12 人标准狼人杀，参照 `werewolf-rules` skill。额外约束：

- 角色分配必须随机且均衡（4狼4民4神）
- 夜晚行动顺序固定：守卫 → 狼人 → 女巫 → 预言家
- 白天发言必须轮流，不能跳过存活玩家
- 投票必须有平票处理机制
- 胜负判定必须正确：屠边规则（狼人杀光所有村民 或 所有神职）
- 警长选举在首日白天进行

## 3. 多语言支持（中/英）

所有用户可见文字必须同时支持中文和英文：

### 实现方式

```python
TEXT = {
    "phase_night": {"zh": "夜晚", "en": "Night"},
    "phase_day": {"zh": "白天", "en": "Day"},
    "role_werewolf": {"zh": "狼人", "en": "Werewolf"},
    "btn_start": {"zh": "开始游戏", "en": "Start Game"},
}
```

- 默认使用中文，可通过配置切换为英文
- 角色名、阶段名、按钮文字、日志消息全部走字典查找，不硬编码字符串
- 语言包放在独立文件或模块中，不在渲染代码里内联文字

## 4. 核心元素可见性

以下元素在**任何分辨率**下必须可见且不被遮挡：

| 元素 | 位置要求 |
|------|---------|
| 阶段指示器 | 顶部居中或左上，始终可见 |
| 玩家列表（12人） | 主体区域，可滚动但不能溢出 |
| 操作按钮 | 底部或侧边栏，始终可点击 |
| 日志/消息 | 侧边栏或底部，最近 5 条可见 |
| 倒计时/天数 | 紧邻阶段指示器 |

### 缩放原则

当窗口小于设计基准时：
1. 先缩小间距和边距
2. 再缩小字体
3. 最后才缩小精灵/图标
4. 不裁剪核心交互元素

## 5. 像素渲染原则

继承 `visual-principles` skill 的风格要求，补充：

- 所有 draw 操作使用整数坐标（像素对齐）
- 缩放使用 `pygame.transform.scale` 配合 NEAREST 邻居采样
- 调色板预先定义，不在渲染循环中创建新颜色对象
- 字体使用 bitmap/像素风格，避免 TrueType 平滑

## 6. 代码组织

- 游戏逻辑与渲染分离：`game/` 目录放纯逻辑，渲染代码在 `renderer.py`
- 配置项（颜色、尺寸、文字）集中管理，不散落各处
- 每个 .py 文件不超过 1500 行
- 关键状态变更必须有日志输出
